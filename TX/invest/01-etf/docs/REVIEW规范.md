# 代码评审规范（REVIEW 规范）

| 项 | 内容 |
| --- | --- |
| 版本 | v1.0 |
| 日期 | 2026-07-10 |
| 适用范围 | 所有 PR / 改动（含 AI 生成的代码） |
| 目标 | 守住松耦合、需求一致、质量与可追溯 |

---

## 1. 评审流程
1. **提交前自检**：作者（含 AI）对照第 3 节清单自查并留痕（见 `AI开发约束.md`）。
2. **提交评审**：描述改动目的、涉及文件、关联 `CHG-` 编号。
3. **评审人核对**：逐条勾选第 3 节清单；不通过则打回并注明原因。
4. **合入条件**：全部 Must 项通过 + 测试通过 + 开发记录已追加。

---

## 2. 评审角色
- **作者**：AI/开发者，负责实现与留痕。
- **评审人**：另一 AI 或人类，负责按本规范核查（松耦合与需求一致性为重点）。

---

## 3. 评审检查清单（Checklist）

### A. 需求一致性（Must）
- [ ] 改动对应明确 SRS 编号（FR-/DR-/IR-/NFR-）。
- [ ] 功能行为符合 SRS 业务规则（如模糊匹配、分位最小样本 250、不适用类 `applicable:false`）。
- [ ] 验收标准（AC-01~10）相关项满足或可验证。

### B. 松耦合（Must，重点）
- [ ] `services/` 未直接 `import akshare` 或任何具体数据源实现。
- [ ] `services/` 仅依赖 `datasource/interfaces` 抽象。
- [ ] 外部 HTTP/网页解析仅出现在 `datasource/*` 与 `core/clients/*`。
- [ ] SQL 仅出现在 `repositories/`。
- [ ] 新增数据源只需新增实现类 + `di.py` 注册，services/api 零改动。
- [ ] 依赖方向单向：`api → services → (interfaces + repositories)`，无反向/跨级。

### C. 改动留痕（Must）
- [ ] `docs/开发记录.md` 已追加 `CHG-` 记录，模板字段齐全。
- [ ] 记录含涉及文件、接口/契约影响、测试情况、回滚方式。
- [ ] 若改接口/契约，已说明兼容/版本化方案。

### D. 错误处理与健壮（Should）
- [ ] 外部源失败有降级或明确 503，不抛裸 500。
- [ ] 参数校验完整（code 6 位、window 合法）。
- [ ] 异常经 `core/errors` 统一封装。

### E. 质量与测试（Must）
- [ ] 新增/修改有对应 pytest，且通过。
- [ ] 含松耦合护栏测试（静态 import 检查）。
- [ ] 无遗留 `print` 调试、无硬编码密钥/域名。
- [ ] 类型注解完整，通过 `mypy`(可选)/`ruff`。

### F. 文档同步（Should）
- [ ] 设计/接口文档随代码同步更新（如路由、字段变更）。
- [ ] 新增依赖写入 `requirements.txt` 并注明用途。

### G. 合规（Must）
- [ ] 出参含 `source` 与 `update_time`。
- [ ] 页面/接口含免责声明。

---

## 4. 评审通过/打回标准
- **通过**：A、B、C、E、G 全部 Must 通过；D、F 无阻断项。
- **打回**：任一 Must 未通过；或发现跨层依赖、未留痕、破坏契约未说明。
- **严重违规**（密钥入库、删除历史记录、伪造测试）：立即终止并上报用户。

---

## 5. 评审记录格式
在 PR/MR 描述或 `docs/开发记录.md` 对应 `CHG-` 条下追加：
```
- 评审人：AI/人类
- 评审结论：通过 / 打回
- 意见：<具体条目与修改要求>
```

---

## 6. 自动化建议
- CI 中跑：`pytest` + `ruff` + 松耦合 import 静态检查（断言 `services/` 不引具体源）。
- 预提交钩子：禁止 `import akshare` 出现在 `services/`、`api/`。
