# AI 开发约束（Development Constraints for AI Agents）

> 适用范围：所有在本仓库中执行代码改动、文档改动的 AI（含接手开发的后续 AI）。
> 目的：保证多人/多 AI 协作可追踪、可回滚、模块松耦合、质量可控。

---

## 0. 总原则
1. **先读后改**：任何改动前，必须先阅读本目录相关文档（至少 `PRD.md`、`SRS.md`、对应设计文档、`编码规范.md`、`REVIEW规范.md`）。
2. **改动留痕**：每一次改动必须按第 3 节格式写入 `docs/开发记录.md`（见第 4 节模板）。
3. **松耦合优先**：严格遵循 `docs/架构设计文档.md` 的分层与依赖规则，禁止跨层直接依赖（详见第 5 节）。
4. **不破坏契约**：不擅自删除/重命名对外接口；确需变更须版本化并在开发记录说明兼容方案。
5. **测试与合规**：改动后相关测试应通过；新增依赖必须登记；不提交密钥。
6. **根因优先，禁止吞错**：遇到报错必须定位并解决**根本原因**，严禁用"捕获后忽略/静默 `pass`/无意义兜底"等方式把报错掩盖过去（详见第 9 节）。
7. **先规划后开发**：开发新功能前，必须先创建 `docs/tasks/task-NN.md` 完成方案规划，评审/自检通过后再进入编码（详见第 10 节）。
8. **BUG 独立留档**：修复 BUG 时，除在 `开发记录.md` 追加 `CHG-` 记录外，还须为该 BUG 单独创建 `docs/bugs/bug-XXX.md`，记录描述、根因、解决办法、影响范围等（详见第 11 节）。

## 1. 改动前必读清单（Checklist）
执行改动前，确认已读：
- [ ] `docs/PRD.md`（产品意图）
- [ ] `docs/SRS.md`（功能/数据/接口需求与验收标准）
- [ ] `docs/架构设计文档.md`（分层与边界）
- [ ] `docs/软件概要设计说明书.md` / `docs/软件详细设计说明书.md`（本次涉及模块）
- [ ] `docs/编码规范.md`
- [ ] `docs/REVIEW规范.md`
- [ ] 若涉及数据源：`docs/数据源调研.md`

---

## 2. 改动类型与要求
| 类型 | 要求 |
| --- | --- |
| 新增功能 | 对应 SRS 的 FR 编号；**先建 `docs/tasks/task-NN.md` 规划**（见第 10 节），再改设计文档、编码、补测试 |
| 修 BUG | 定位并解决**根本原因**（禁止吞错，见第 9 节）；**新建 `docs/bugs/bug-XXX.md`**（见第 11 节）记录描述/根因/解决办法/影响范围/复现条件 |
| 重构 | 必须保持对外接口不变；说明松耦合收益 |
| 换数据源 | 仅在 `datasource/` 层内；不得影响 `services/` 与 `api/` |
| 改接口/契约 | 版本化（如 `/api/v2/...` 或字段兼容）；开发记录注明迁移方案 |
| 文档更新 | 同步更新受影响的设计/接口文档，保持一致性 |

---

## 3. 改动留痕规则（强制）
- **唯一记录文件**：`docs/开发记录.md`。
- **每次改动**（一次完整任务/一次会话的连续改动）追加**一条**记录，不得覆盖历史。
- **编号**：`CHG-YYYYMMDD-NN`（同日多条递增）。
- **内容模板**见第 4 节，缺一不可填（无则写"无"）。
- 涉及多文件时，列出全部文件相对路径。
- 若改动影响数据库 schema，必须注明迁移方式（新增/ALTER/重建）。

```markdown
- 关联需求：SRS FR-03 / PRD F-03

  - src/datasource/interfaces.py（新增抽象）
  - src/datasource/akshare_src.py（新增实现）
  - tests/test_akshare_src.py（新增测试）

  - before：无估值数据源
  - after：ValuationSource 接口 + AkShare 实现，返回 pe_ttm/pb

```

## 5. 松耦合硬性约束
1. **分层依赖单向**：`api → services → (datasource 接口 + repositories)`；禁止反向或跨级依赖。
2. **外部依赖隔离**：`akshare`、HTTP 请求、网页解析只允许出现在 `src/datasource/` 与 `src/core/clients/`；`services/`、`api/` **严禁**直接 `import akshare` 或发请求。
3. **依赖倒置（DIP）**：`services/` 只依赖 `datasource/interfaces.py` 中定义的抽象类；具体实现通过依赖注入（DI）传入。
4. **模块边界**：每个模块对外只暴露明确定义的接口/函数；内部实现（私有函数、具体类）不对外。
5. **配置外置**：数据源开关、更新频率、超时等放 `config`，不在代码硬编码。
6. **数据访问封装**：所有 DB 读写经 `repositories/`，业务层不直接写 SQL。
7. **可替换性验证**：更换一个数据源实现，不应修改 `services/` 与 `api/` 任何代码（仅改注入配置）。

---

## 6. 依赖与环境
- 新增 Python 依赖必须写入 `requirements.txt` 并注明用途；禁止安装无关包。
- 使用虚拟环境（venv/conda）。
- 密钥/Token 走环境变量或 `.env`（不入库）；`.env` 已在 `.gitignore`。

---

## 7. 提交与版本
- 不自动 `git commit`，除非用户明确要求。
- 需要提交时，提交信息引用 `CHG-` 编号，如 `CHG-20260710-01 实现估值适配层`。
- 重大版本变更同步更新 `CHANGELOG.md`（发布级，区别于开发记录）。

---

## 9. 根因修复原则（禁止"吞错"）
1. **先定位再修复**：遇到报错/异常/测试失败，必须先复现并查明**根本原因（root cause）**，再动手修复；不得凭猜测改代码。
2. **禁止掩盖错误**，以下均视为违规：
   - `except ...: pass` 或 `except Exception: ...` 后不处理、不记录、不重抛，仅为让程序"跑过去"；
   - 用空 `try/except` 包住报错点却不解决实际问题；
   - 直接注释掉报错代码、下调断言/校验、`# type: ignore`、`# noqa` 等方式绕过而非解决；
   - 用无意义的默认值/兜底静默吞掉本应暴露的错误。
3. **允许的异常处理**：确有业务理由的兜底（如外部数据源不可用时降级），必须**同时满足**——① 有明确注释说明为何可容忍；② 记录日志（含上下文）；③ 不掩盖调用方需要感知的错误。
4. **留痕**：根因与修复方式写入对应 `CHG-` 记录；若为 BUG，另写 `docs/bugs/bug-XXX.md`（第 11 节）。

## 10. 新功能任务规划（`docs/tasks/task-NN.md`）
1. **时机**：开发任何新功能**之前**先创建，规划评审/自检通过后再编码。
2. **命名**：`docs/tasks/task-NN.md`（`NN` 递增，两位数字，如 `task-01.md`）。
3. **模板**：
```markdown
# task-01：<功能名称>
- 创建人：AI（CodeBuddy）
- 日期：YYYY-MM-DD
- 关联需求：SRS FR-xx / PRD F-xx
- 状态：规划中 / 开发中 / 已完成

## 1. 目标与范围
<要解决的问题、边界、非目标>

## 2. 方案设计
<涉及的分层/模块、接口定义、数据结构、DB 变更、依赖>

## 3. 实施步骤（拆解）
- [ ] 步骤1
- [ ] 步骤2

## 4. 测试计划
<单元/集成用例、验收标准>

## 5. 风险与回滚
<潜在风险、依赖波动、回滚方式>
```
4. **联动**：编码完成后在 `开发记录.md` 追加 `CHG-` 记录，并回填本 task 的"状态"为已完成。

---

## 11. BUG 记录（`docs/bugs/bug-XXX.md`）
1. **时机**：修复 BUG 时创建，与 `开发记录.md` 的 `CHG-` 记录并存（互相引用）。
2. **命名**：`docs/bugs/bug-XXX.md`（`XXX` 递增，三位数字，如 `bug-001.md`）。
3. **模板**：
```markdown
# bug-001：<BUG 简述>
- 记录人：AI（CodeBuddy）
- 日期：YYYY-MM-DD
- 关联记录：CHG-YYYYMMDD-NN
- 严重级别：低 / 中 / 高 / 阻断

## 1. BUG 描述
<现象、错误信息、期望 vs 实际>

## 2. 复现条件
<环境、步骤、输入>

## 3. 根本原因（Root Cause）
<定位过程与真正原因，非表象>

## 4. 解决办法
<具体修复方案、涉及文件>

## 5. 影响范围
<受影响的模块/接口/数据/用户场景；是否需数据修复/迁移>

## 6. 验证
<修复后的验证方式与结果、回归测试>
```

---

## 12. 违规处理
若 AI 违反上述约束（如跨层依赖、未留痕、破坏接口、吞掉报错、新功能未先规划、修 BUG 未建档）：
- 在 REVIEW 阶段必须被拦截并回退；
- 严重违规（如泄露密钥、删除历史记录）立即终止并报告用户。
