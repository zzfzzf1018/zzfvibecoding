## 1. 架构目标与质量属性
| 质量属性 | 目标 | 设计策略 |
| --- | --- | --- |
| 松耦合 | 换数据源/换 DB 不影响业务与接口层 | 分层 + 依赖倒置（DIP）+ 适配器模式 |
| 可替换性 | 主源故障切兜底源，零代码改动 | 多实现 + DI 注入 + 调度降级 |
| 可测试性 | 单测不依赖外网 | 接口抽象 + Mock 适配 |
| 可观测 | 数据新鲜度/失败可监控 | 统一日志 + `update_time` + 告警 |
| 性能 | 命中缓存检索 ≤ 500ms | 本地缓存库 + 分页 |
| 可维护 | 新增源成本低 | 模块化、配置化 |

## 2. 架构风格
采用**分层架构 + 端口与适配器（六边形）思想**：
- 核心业务（services）位于中心，只依赖抽象端口（interfaces）。
- 外部（数据源、DB、HTTP、UI）作为适配器挂在端口上。
- 入站端口：HTTP API / UI；出站端口：ValuationSource / ConstituentSource / EtfBasicSource / Repository。

```
┌──────────────────────────────────────────────┐
│                  UI (Web / Streamlit)          │  入站适配器
└───────────────┬──────────────────────────────┘
                │  HTTP
┌───────────────▼──────────────────────────────┐
│              API 层 (FastAPI routers)          │  入站端口
└───────────────┬──────────────────────────────┘
                │ 调用
┌───────────────▼──────────────────────────────┐
│        Services 业务层 (检索/估值/分位/成分股)   │  核心（只依赖接口）
│  依赖：datasource.interfaces + repositories      │
└───────┬───────────────────────────┬───────────┘
        │ 端口（抽象）                │ 端口（抽象）
┌───────▼────────┐          ┌────────▼──────────┐
│  DataSource     │          │   Repositories     │  出站适配器
│  Adapters       │          │   (DB 缓存/存储)   │
│ (akshare/csindex│          │   (SQLAlchemy)     │
│  /em impl)      │          │                    │
└───────┬────────┘          └────────┬──────────┘
        │ 外部 HTTP/网页               │ SQLite/PostgreSQL
┌───────▼────────┐          ┌────────▼──────────┐
│ 外部数据源       │          │  本地数据库          │
│ (中证/东财/新浪) │          │                    │
└────────────────┘          └───────────────────┘

┌──────────────────────────────────────────────┐
│  Scheduler 定时任务：每日拉取并写入 Repositories   │
└──────────────────────────────────────────────┘
┌──────────────────────────────────────────────┐
│  Core：config / logging / errors / di（横切关注点）│
└──────────────────────────────────────────────┘
```

## 3. 模块划分与职责
| 模块 | 职责 | 依赖（仅向下） |
| --- | --- | --- |
| `app/api` | HTTP 路由、参数校验、响应封装 | services |
| `app/services` | 业务逻辑：检索、估值聚合、分位计算、成分股聚合 | datasource.interfaces, repositories |
| `app/datasource/interfaces` | 抽象端口（ValuationSource 等） | 无 |
| `app/datasource/akshare_src` | AkShare 实现（主） | interfaces, core/clients |
| `app/datasource/csindex_src` | 中证官网实现（兜底） | interfaces, core/clients |
| `app/datasource/em_src` | 东方财富实现（兜底） | interfaces, core/clients |
| `app/repositories` | 数据库读写（缓存/历史） | models, core |
| `app/models` | ORM 模型 + Pydantic 出参 Schema | 无 |
| `app/scheduler` | 定时更新估值/成分股/基础信息 | services, repositories |
| `app/core` | config、logging、errors、DI 容器 | 无 |

---

## 4. 依赖规则（松耦合核心）
1. **单向依赖**：`api → services → (interfaces + repositories)`。禁止 `services` 反向依赖 `api`，禁止 `datasource` 依赖 `services`。
2. **依赖倒置**：`services` 仅 `import` 自 `datasource.interfaces`；具体类（akshare_src 等）在 `core/di.py` 组装后注入。
3. **外部隔离**：`akshare`/`requests`/网页解析只允许在 `datasource/*` 与 `core/clients/*`。
4. **数据访问封装**：SQL 仅出现在 `repositories/`；`services` 调仓储方法而非拼 SQL。
5. **横切复用**：日志/配置/异常统一走 `core/`，各层不重复造轮子。

---

## 5. 数据架构
- 存储：MVP 用 SQLite；生产可换 PostgreSQL（仅改 repository 连接串，不动逻辑）。
- 三大数据集（见 SRS DR-01~05）：`etf_basic`、`index_valuation`、`index_valuation_history`、`index_constituent`。
- 关联键：`etf_basic.track_index_code → index_*` 的 `index_code`。
- 缓存策略：基础信息 24h、估值 T+1、成分股 7d/公告触发；外部请求先查库，未命中再请求并回写。

## 6. 关键设计决策（ADR 摘要）
| 决策 | 选型 | 理由 |
| --- | --- | --- |
| 语言/框架 | Python + FastAPI | AkShare 为 Python；FastAPI 异步友好、自动文档 |
| 数据源主源 | AkShare | 免 token、覆盖广、活跃 |
| 解耦方式 | 适配器 + DI | 满足松耦合、可替换 |
| 历史序列 | 自采落库 | 第三方全量历史不稳，自管可控 |
| MVP UI | Streamlit（可选） | 快速验证全链路 |

## 7. 部署视图（建议）
```
[ 浏览器 ] ──HTTPS──► [ FastAPI (uvicorn) ]
                         │
          ┌──────────────┼───────────────┐
          ▼              ▼               ▼
     [Scheduler]    [SQLite/PG]     [出站: 数据源 HTTP]
```
- 单机即可运行；MVP 无需独立中间件。
- 定时任务进程或同进程 APScheduler；外网出口需放行中证/东财/新浪域名。

---

## 8. 风险与缓解（架构层面）
- 数据源限流 → 缓存 + 请求间隔 + 重试（core/clients）。
- 主源失效 → 多适配器 + 调度降级（services 选源逻辑）。
- 跨层耦合漂移 → REVIEW 规范强制检查 import 边界。
